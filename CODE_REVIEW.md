# DspUniversalDepot Code Review (v0.2.3 → v0.3.0/1.0.0)

Reviewer: Hermes Agent (audit). Repo: /home/hermes/DspUniversalDepot/

## CRITICAL BUGS (would break the mod, must fix)

1. **Duplicate Harmony patches with conflicting semantics** — `StorageComponent.TakeItem` is patched **twice**:
   - `src/RecipePatcher.cs:91-108` (`StorageComponent_Patch.TakeItem_Prefix`) — applies to *every* entity unconditionally and returns `false` (skip original), stealing ALL takes from non-depot storages.
   - `src/ConveyorPatcher.cs:75-91` (`PatchStorageQueries.TakeItem_Prefix`) — does the right thing with `IsUniversalDepot` check.
   These two prefixes cannot coexist; whichever Harmony binds first will steal every `TakeItem` call, **breaking vanilla logistics vessels, labs, assemblers, miners, ILS, etc.**. The RecipePatcher version has no depot guard and no `IsUniversalDepot` filter. **This is a hard showstopper for v0.3.0.** Delete the duplicate in `RecipePatcher.cs` (the whole `StorageComponent_Patch` class, lines 85-109).

2. **Duplicate VFPreload patch — both `LDBPatcher.cs:156-172` and `RecipePatcher.cs:44-61` patch `VFPreload.InvokeOnLoadWorkEnded`** with two different Postfixes calling `LDBPatcher.ApplyPatches()` (or `AddBuilding`+`AddRecipe`). The postfixes will fire in unspecified order; `ApplyPatches()` is called twice. That is a correctness *and* a runtime cost bug. Pick one site; the `LDBPatcher.cs` one is the right home.

3. **Multiple `new Harmony(GUID)` instances** — `UniversalDepotPlugin.Load()` creates one (line 108), `ConveyorPatcher` ctor creates another (line 31), `LDBPatcher.Register()` creates a third (line 39). All three call `PatchAll(...)` on the same assembly. That double/triple-patches the same methods and leaves Harmony instance objects alive forever (no `UnpatchAll` on plugin unload). One Harmony instance, one `PatchAll`. Drop the others.

4. **`IsUniversalDepot` returns true for any entity that has *any* storage ever patched** — `ConveyorPatcher.cs:119-126` only checks `Storage.Contains(entityId)`. But `Contains` is only true after `GetOrCreate` has been called once, and that only happens *inside* a Harmony prefix. There is a chicken-and-egg problem: the first time a belt queries the depot, `GetOrCreate` is called → `Contains` becomes true → future queries go to the depot path. *However* if the player ever places a vanilla `StorageComponent` (e.g. another mod adds a vanilla-style depot at the same entityId, or the player reuses the entity after deletion), we will steal that storage. More importantly, the first ever query of a Universal Depot (e.g. via `GetItemCount` before any `AddItem`/`TakeItem`) **silently returns the vanilla value** because `GetOrCreate` only fires in the Prefix when an item is actually present, and the dispatcher itself is downstream of the entityId-is-depot check. Verify with a manual test or use `EntityData.protoId == CustomItemId.Value` (real DSP API) as the gate.

5. **Version mismatch between `manifest.json` (0.2.3) and `DspUniversalDepot.csproj` / `UniversalDepotPlugin.cs` (still 0.2.2)** — `manifest.json:3` says `0.2.3`, but `src/UniversalDepotPlugin.cs:21` and `src/DspUniversalDepot.csproj:11` both say `0.2.2`. The plugin log will print `DspUniversalDepot v0.2.2 loaded` while Thunderstore reports 0.2.3. Bump both source-of-truth files to 0.2.3 (or, if releasing v0.3.0 next, do a version-only commit).

6. **In-game recipe is registered as a stub** — `LDBPatcher.cs:51-67` (`ApplyPatches`) is called from the VFPreload postfix, but its four bodies (`EditItemsData`, `EditRecipesData`, `EditStringBuilder`, `EditProto`, `EditBuildIndex`) are **all stubs that only log**. The building is never actually added to `LDB.items`, the recipe never actually added to `LDB.recipes`, no `ItemProto` is registered, no build index. The depot will be **completely unbuildable in the real game**; the player won't see it in the build menu, can't craft it, and even if they hack one in via console the patches above will fire on a vanilla `StorageComponent` (per bug #1). This is the largest gap between the README and the code. **For v0.3.0/1.0.0 you must implement the LDBTool API calls** (`LDBTool.EditDataAction += ...` + `LDBTool.PreAddDataAction += ...` plus `ProtoRegistry` if you go that route).

## BUGS (wrong behavior, should fix)

1. **`StorageManager` `AddItems` "oldest item" semantics inverted** — `StorageManager.cs:108-112` and `:168-184`. When a slot is full and `DeleteOverflow=true`, `EvictOldestItems` picks the slot with the *smallest* `DateTime.UtcNow` value. The very slot being added to has just been touched (line 112), so it's now the *newest* — good, it is excluded by the `excludeItem` filter. *But* `AddItems` runs `_slotTimestamps[itemId] = DateTime.UtcNow` BEFORE checking capacity, so on overflow the "oldest" slot is the one least recently added to, which is what you want. ✅ Correct after re-reading. Marking this item as "looks suspicious, but actually OK on close inspection."

2. **`TakeItems` does not update `_slotTimestamps`** — `StorageManager.cs:134-142`. Removing items is invisible to the overflow policy. If a slot was emptied by conveyor pulls, the timestamp stays old, so on the next overflow it's the first candidate for deletion. That may be intentional ("oldest touched") but combined with the Add path that always touches, it means a slot can be protected purely by being read from. Probably want to `DateTime.UtcNow` only when the slot is being filled above a threshold, or document the policy.

3. **`StorageManager.AddItems` returns the *rejected* count, not added** — `StorageManager.cs:127`. Caller in `ConveyorPatcher.cs:108-110` correctly does `__result = count - rejected`, so end result is OK. But the API is non-idiomatic ("returns 0 = all added" is the docstring, so the contract is actually consistent: return value is "amount not fit"). Add a `# XML doc` comment and/or rename to `TryAddItems`/`AddItemsReturningRejected`.

4. **FirstOrDefault on a `Dictionary<int, …>` returns `0` as "missing"** — `StorageManager.cs:175-176`. If the oldest slot happens to have itemId `0` (which is reserved in DSP for "no item"), `EvictOldestItems` short-circuits with `if (oldest == 0) return 0;`. This is a silent bug — when itemId 0 leaks in, eviction never happens. Use a sentinel (`int oldest = -1; ... ; if (oldest < 0) return 0;`).

5. **LinqShim is dead code** — `StorageManager.cs:188-202`. You `using System.Linq;` at the top, so the shim's extension methods (with the same signatures) are *also* defined, but the System.Linq ones win at compile time. The shim is never invoked. Either remove it, or remove `using System.Linq;` and rely on the shim (the file comment claims Unity 2022 = C# 9, but .NET 6 = C# 10, so System.Linq is always there). Dead code = maintenance burden.

6. **`OrderBy().First()` is O(n log n) per call but `_slotTimestamps` is small** — `StorageManager.cs:157-159, 170-174`. Functional, but a `LinkedList<int>` keyed by insertion order or a single `DateTime oldestTs` field would be O(1). Premature optimization, but the comment "Unity 2022 = C# 9 compatible" implies this was hand-rolled to avoid LINQ. Now that System.Linq is imported, the shim is dead anyway (see #5).

7. **`AssetBundleManager.Load()` looks next to the DLL** — `AssetBundleManager.cs:28-29`. With BepInEx the canonical location is `BepInEx/plugins/DspUniversalDepot/universaldepot.assets` and the DLL lives in the same folder, so this works. But many users install via r2modman which nests `BepInEx/plugins/DspUniversalDepot/DspUniversalDepot.dll` inside a subfolder. The `manifest.json:6` says `icon = icon.png` (root of zip), so r2modman will pull `universaldepot.assets` only if it is at zip root or under `plugins/...`. Add a fallback: also try `Path.Combine(pluginPath, "..", "universaldepot.assets")` and the parent dir, log a clearer message.

8. **No save/load integration** — there is no `DSPModSave` save/load hook (the reference is in the .csproj but never used). After a save → quit → reload cycle, **all Universal Depot contents are lost**, but the building still exists in the world and will keep creating `DepotStorage` entries on demand. Slots are effectively empty. This is a fatal UX bug for a "universal storage" mod: the entire point is the stuff inside it. Add `DSPModSave.OnGameDataExport` / `OnGameDataImport` handlers writing the `_depots` dictionary (itemId → count) to the save file.

9. **No cleanup on planet unload / star destroy** — `StorageManager.Remove(int)` exists (line 31) but **no caller exists anywhere in the codebase**. When a planet is destroyed (or the player leaves the cluster and DSP unloads the `PlanetFactory`), the `StorageComponent` instance is GC'd but our `_depots[entityId]` lives forever. Over a long playthrough this leaks one `DepotStorage` (and its `Dictionary<int,int> + Dictionary<int,DateTime>`) per destroyed Universal Depot. Patch `PlanetFactory.RemoveEntityData` (or DSP's "entity destroyed" event) and call `Storage.Remove(entityId)`.

10. **`IsInputLane/IsOutputLane` are not used anywhere** — `ConveyorPatcher.cs:39-44`. The constants are exported but no caller enforces the "no ILS ports" claim. The Harmony patches intercept every `StorageComponent` call regardless of port — so if a player plugs an Interstellar Logistics Station into a Universal Depot (e.g. by mis-wiring), it will *probably* still work because both sides funnel into the same `DepotStorage`, but the README claim "0 ILS ports" is not enforced in code.

11. **`new StorageManager()` is created in `Load()` but `StorageManager.AddItems` / `TakeItems` are called before the player has placed any depot** — `UniversalDepotPlugin.cs:112`. The first belt call to `GetItemCount` on a *vanilla* entity will not match `IsUniversalDepot` (returns `false`) — good, it falls through. *But* the very first call on a newly-placed Universal Depot (entityId not yet in `_depots`) will: `Contains(entityId)` returns `false` → returns `true` (fallthrough to vanilla) → vanilla `StorageComponent.GetItemCount` returns 0 because no DSP slot exists for this entity. The depot is then placed, but until the first `AddItem` happens the count is always 0, and the *first* `AddItem` will work because the prefix's `GetOrCreate` is called inside `AddItem_Prefix` which has no `IsUniversalDepot` filter in the AddItem path's caller. Net effect: works by accident, but a strict reader would expect `IsUniversalDepot` to actually be a positive ID check (`entityData.protoId == CustomItemId.Value`) rather than a "we already touched this entity" check.

12. **`#if !HAS_DSP_REFS` guard on the whole `_Stubs.cs` is correct for "stubs-only mode" but the file is compiled in real-DLL builds in CI** — `src/_Stubs.cs:4`. In the real-DLL build (`!HAS_DSP_REFS` is false), `_Stubs.cs` is empty. ✅ Correct. But the comment at the top of `_Stubs.cs:1-3` says "ONLY compiled when DSP_GAME_PATH is NOT set", which is misleading: the real condition is `!HAS_DSP_REFS` which is set in both the `DSP_USE_STUBS=true` *and* the `DSP_GAME_PATH` paths. The comments lie. Tighten them.

13. **`StorageComponent_Patch` in `RecipePatcher.cs` has *two* `TakeItem` patches with different overloads** — `src/RecipePatcher.cs:92` specifies `Type[] { int, int, int, int }`, but the real DSP `StorageComponent.TakeItem` has many overloads. The other prefix in `ConveyorPatcher.cs:76` uses `[HarmonyPatch(typeof(StorageComponent), "TakeItem")]` without the argument array — Harmony will pick the no-arg overload (or throw "no method found") and silently fail. Pick the right overload signature for DSP's `StorageComponent.TakeItem(int, int, int, out int)` (the real signature returns by `out` parameter, not by `ref int __result`). **This is likely a runtime crash** when the prefix is bound: ref `__result` ↔ out arg mismatch.

14. **`PatchAll` is called three times on the same assembly** — see CRITICAL #3. Harmony will apply the same patch three times unless the second and third `PatchAll` calls are no-ops (they're not — `PatchAll` does not dedupe).

15. **`StorageComponent_Patch.TakeItem_Prefix` parameter `ref int __result` is wrong** — `src/RecipePatcher.cs:99`. The real DSP `StorageComponent.TakeItem` signature is something like `int TakeItem(int, int, int, out int)` — i.e. it returns the count *and* sets an `out int` for the remainder. `ref int __result` on a method whose return is not `int` (it's `int` here, so the prefix *must* skip the original and return `__result`) — but the method already returns int, so Harmony will not let you write `ref int __result`. Compile-time check on a stub passes because the stub returns `int`; runtime will fail when Harmony's IL-level matching looks for a matching signature. **Verify with a real DSP build.**

## IMPROVEMENTS (would be nicer but not breaking)

- **Unify all `Log` access** through a single `BepInEx.Logging.ManualLogSource` alias; the `new` keyword or rename fixes the CS0108 warning (`UniversalDepotPlugin.cs:24`).
- **Move the Harmony instance** to a private field (`private Harmony _harmony;`) so `UnpatchAll` can be called from a future `Unload()` override (BepInEx IL2CPP has `Unload` too).
- **Bake the "Universal Depot" check** with `GameMain.data.factories[planetId].entityPool[entityId].protoId == CustomItemId.Value` — single source of truth, no "we have a DepotStorage" chicken-and-egg.
- **Add a `BuildIndex` collision check** on startup: if `LDB.items` already contains an item with the user's `CustomItemId`, log a fatal error and refuse to register.
- **Add a `LinqShim` removal commit** — it's dead code.
- **Add `try/catch` around each `LogInfo`** in the load method (defensive — Log source might be null if a hot-reload happens).
- **Cap `ItemLimit` and `MaxSlotCount`** at sane upper bounds in `Config.Bind` validators — currently nothing prevents a user setting `ItemLimit=999999999` and OOMing the dictionary's value-type churn.
- **Add an `info`/`warn` log on `StorageManager.Clear()`** (currently logs even on empty) — `StorageManager.cs:42-46` always logs "All depots cleared" regardless of state.
- **Add a `[HarmonyPatch(typeof(...), nameof(...))]` to disable a method after the patch binds** — currently `UniversalDepotPlugin.Load` does not check whether `harmony.PatchAll` succeeded.
- **Add a `Settings` sub-section in the config** for "true"/"false" reset-to-default button (BepInEx supports this via `AcceptableValueList`/`ConfigFile.SettingChanged`).
- **Replace `Dictionary<int, int> _slots` with `List<DepotSlot>`** where each slot has `int ItemId`, `int Count`, `DateTime LastTouched` — one allocation, less GC churn.
- **Use `[Description]` attributes on the config entries** so they show tooltips in r2modman.
- **In `AssetBundleManager.Load`, wrap `Bundle.LoadAsset<T>` calls in `Bundle.GetAllAssetNames()` checks** so a malformed bundle logs precisely which asset is missing.
- **In `download-libs.sh`, pin the BepInEx version in the URL** — currently hard-coded to `5.4.23.5`, fine, but if BepInEx republishes the asset the script breaks silently. Add a SHA-256 check or `-f` to fail fast.
- **CI `build.yml` does `continue-on-error: true`** for the build step — meaning a broken build still passes. Replace with a proper stub build target (you already have the `#if !HAS_DSP_REFS` path; just trigger it with `-p:DSP_USE_STUBS=true`).
- **Add a `releases/DspUniversalDepot-0.2.3-source.zip`** to the release workflow — currently it only uploads the source zip; the r2modman zip is committed manually.
- **`Tools/AssetBundleBuilder.cs` is in `tools/` but `BUILDING.md` says to copy it into `Assets/Editor/`** — add a one-liner script that does that copy.
- **Add a `BepInEx/` placeholder file** or a no-op `BepInEx.dll` so the r2modman zip doesn't fail the "BepInExPack is a dependency" check (some validators require the folder to exist).
- **Add `package_version_strategy: ^` or `>=` in the manifest** — currently `version_number: 0.2.3` is a single version, fine for Thunderstore.

## UX / BALANCE ISSUES

- **Recipe cost is opaque**: 30 Ti + 20 CB + 10 Micro + 2 BB at 10s is ~equivalent to **2 Interstellar Logistics Stations** worth of material (each ILS = 12 Ti + 12 CB + 12 Micro + 1 BB at 1s in vanilla). 10s for 1 depot = 0.2 ILS-equivalent/sec, so the recipe is **roughly 10× the materials of one ILS for half the output rate** — actually *under*-priced for a building that gives unlimited slots. The README claim "expensive to balance unlimited storage" doesn't survive the math. **Consider** 60 Ti + 40 CB + 20 Micro + 4 BB at 20s, or just price it 3× the current recipe. (Aside: real DSP has the 2 BB produced by particle colliders late-game, so 2 BB is a meaningful late-game cost — actually fine.)
- **ItemLimit 5000 is ~2× a fully-loaded MK3 belt per second.** With 3 belt lanes at 30 items/sec, the depot fills in 5000/(30*3) ≈ 55 seconds before refusing. For an "unlimited" item mod, this default is low. **Consider 50000 default** (15+ min to fill at peak throughput).
- **MaxSlotCount 1000 is a soft cap; real DSP has 100+ items in a playthrough.** Fine, but combined with `DynamicSlots=true` and 5000 stack, a single depot holds 5M items — more than the universe generates in a normal play. The whole "unlimited items" promise feels already-met.
- **DeleteOverflow: no confirmation dialog, no chat warning** when the 1st item is auto-deleted. Players will lose items and not know why. Suggest: chat-bar message on first delete per save, and a config-driven `LogWarning` always.
- **No UI to inspect a Universal Depot's contents.** Vanilla DSP has a UI panel that shows the item grid; since you're intercepting the data, the existing UI *might* show "0 items" or freeze. Need to test in-game — if broken, a custom `UIStorageWindow` would be a v1.0 must-have.
- **Three belt lanes per direction = 6 belts per building.** ILS in vanilla has 4 ports (2 in / 2 out) because the ILS uses 2 of those slots for the ILS remote logistics. The Universal Depot uses 6 for belts — visually a busy building. Might want a "BeltLaneCount" config (default 3) so the player can drop it to 2 for compactness.
- **`IsInputLane` / `IsOutputLane` are dead code** — either wire them into port validation, or delete them. They imply a feature that doesn't exist.

## ADDITIONAL NOTES (not bugs, but worth flagging)

- **`.gitignore` lists `build/` and `libs/`, but `releases/DspUniversalDepot-0.2.3-r2modman.zip` is tracked.** Two untracked `0.2.1`/`0.2.2` zips in `releases/` should be deleted (or `.gitignore` should exclude them and only `0.2.3` should be in the repo).
- **`build/DspUniversalDepot/...` is an artifact folder left over** from a previous `cp -r` or build script. Add to `.gitignore` and clean.
- **`build.yml` workflow is fundamentally broken for verifying the real build** — it builds with stubs (`-p:DSP_USE_STUBS=true` would be the correct path, but the workflow hacks the .csproj with `sed`). Replace the sed hackery with a single `dotnet build -c Release -p:DSP_USE_STUBS=true`.
- **The `CSPROJ` ships `AssemblyPublicizer.MSBuild` in `<NoWarn>` (line 13)** — good, but the corresponding `BepInEx.AssemblyPublicizer.MSBuild` package is downloaded in `download-libs.sh` only to be filtered out (line 71). Cleaner to not download it.
- **`NoWarn` masks CS0103, CS0117, CS0029 etc.** — i.e. if a real DSP build ever references something that doesn't exist on the reference DLLs, you will get silent failures. Tighten `NoWarn` to only the stub-specific codes (CS0414, CS0169, CS0649, CS0067) and let real build errors bubble.
- **`manual_log_source` and the `Log` shadow** — BepInEx's `BasePlugin.Log` is a property; you shadow it with a public static field of the same name. The CS0108 is the only warning, but it changes the call site for code that *was* using `base.Log`. Fix with `new public static ManualLogSource Log;` (intentional hiding).
- **`Download-libs.sh` does not pin DSP modding package versions** — every fresh run may pull a newer (breaking) `CommonAPI.dll` or `NebulaAPI.dll`. Add a version pin in the `for pkg in ...` loop, e.g. `"DysonSphereProgram.Modding.CommonAPI:1.2.3"`.
- **No LICENSE attribution for vendored DSP API knowledge** — fine for a mod, but if you ever copy code from a DSP decompilation project, attribute it.
- **`BepInEx.Harmony` reference and `0Harmony` reference** are both included in the .csproj (lines 57-68). You only need one; pick the one BepInEx actually loads (`0Harmony.dll` is the runtime HarmonyX2 in BepInEx 5.4).

## OVERALL VERDICT

**This is still firmly alpha, not v1.0.** The most important things in the README — *a craftable building, a real recipe, a real 3D model, real LDB integration* — are all **stub log lines in the source code**. The mod will compile, will load via BepInEx, will print a happy "Universal Depot registered" log, and then **the player will find nothing in the build menu and will not be able to craft the building**. The "universal" storage system (`StorageManager.cs`) is real and well-thought-through, and the dynamic slot / overflow logic is solid, but it's protecting a building that doesn't exist. Save/load, planet destroy cleanup, and the duplicate-patch / wrong-overload bugs are all `v0.3.0 must-fix`. The CSV/manifest packaging, CI, and CI release flow are good *plumbing*; what they package is a mod that does ~30% of what the README promises. **Recommend tagging v0.3.0 only after**: (1) deleting the duplicate Harmony patches and unifying the `StorageComponent` prefix with the right overload signature, (2) implementing real LDBTool/ProtoRegistry calls in `LDBPatcher.ApplyPatches`, (3) wiring `DSPModSave` for persistence, (4) fixing the version mismatch, (5) testing in a live DSP instance to confirm the building appears and is craftable, (6) deciding whether `IsUniversalDepot` should consult `EntityData.protoId` or stay as the current "is in our dict" check. **Do not publish to Steam Workshop or Thunderstore with the current code**; the first user review will be "the mod doesn't do anything".
